**1 执行摘要**

AgentCommander 是一款面向开发者的一站式多 AI
智能体协作指挥平台，旨在通过可视化管理界面统一调度和协调多个 AI
编码智能体（如 Claude Code、Cursor、Windsurf
等）协同工作。平台的核心价值在于将分散的 AI
工具整合为一个统一的指挥中心，使开发者能够高效地分配任务、监控进度、收集反馈，从而实现真正的\"AI
团队\"协作开发模式。

随着 AI 编码工具的快速发展，开发者面临着日益复杂的工具链管理挑战。单个
AI
智能体虽然能力强大，但在面对大型、多模块的项目时，缺乏全局视野和协作能力。AgentCommander
通过引入任务分解、依赖管理、并发控制和实时监控等机制，有效解决了这一痛点。平台不仅支持手动管理模式（开发者手动分配任务、审批操作），更提供了自动执行模式（基于
Feedback Loop 的自主任务分发与自测试闭环），极大地提升了开发效率。

本产品的核心差异化优势体现在三个维度：第一，支持多智能体的隔离与协作------每个
Agent 拥有独立的执行上下文和 tmux 会话，同时通过通信总线实现跨 Agent
协作；第二，深度集成 Claude Code Hooks 系统，实现对 AI
工具行为的实时监控与事件收集；第三，基于文件的持久化规划模式（planning-with-files），确保任务状态和执行日志的可追溯性与可恢复性。

目标用户群体包括：个人全栈开发者（利用 AI
加速独立项目开发）、技术团队负责人（管理多个 AI
辅助的编码任务）、开源项目维护者（自动化处理 Issue 和 PR）、以及 AI
应用研究人员（实验和对比不同 Agent 的表现）。AgentCommander 的愿景是成为
AI
协作开发领域的基础设施级平台，让每一个开发者都能像指挥一支高效团队一样驾驭
AI 编码智能体。

**2 产品概述与愿景**

**2.1 背景与动机**

2024-2026 年间，AI
编码智能体经历了从简单代码补全到自主项目开发的革命性跃迁。Claude
Code、Cursor、Windsurf、GitHub Copilot Workspace
等工具纷纷涌现，各自具备独特的优势。然而，开发者在使用这些工具时面临几个关键问题：首先，不同工具之间缺乏统一的调度接口，切换成本高；其次，单个
Agent 在处理复杂任务时容易出现上下文溢出、方向偏离等问题；第三，多 Agent
协作时的文件冲突、任务依赖和进度同步等挑战尚未得到系统化解决。

与此同时，auto-coding-agent-demo 项目展示了 AI 在 10
小时内从零开始自主完成编码项目的惊人潜力，证明了全自动编码的可行性。而
planning-with-files 项目则提供了一种基于文件系统的持久化规划方案，使 AI
能够通过 process.txt、worklog.md
等文件实现跨会话的上下文保持。这些实践为 AgentCommander
的设计提供了重要的灵感来源。

**2.2 产品定位**

AgentCommander 定位为\"AI 智能体的指挥与协作平台\"（AI Agent
Orchestration Platform），而非又一个 AI
编码工具。它不直接生成代码，而是作为上层编排系统，管理、调度和监控底层各种
AI 编码智能体的执行过程。类比于 Kubernetes 之于容器编排，AgentCommander
致力于成为 AI
智能体编排领域的标准平台。产品采用本地优先（Local-First）的架构设计，所有数据存储在用户本地，确保代码安全和隐私保护，同时为未来的云协作和团队共享能力预留架构扩展空间。

**2.3 目标场景**

-   全自动化编码：给定一份 PRD
    或需求文档，系统自动将其拆解为任务队列，并调度多个 AI Agent
    逐步完成开发，实现从需求到代码的全链路自动化。

-   团队模拟开发：模拟真实软件开发团队的角色分工（产品经理、前端工程师、后端工程师、测试工程师等），用不同配置的
    AI Agent 扮演不同角色协同开发。

-   技术研究与原型验证：快速启动多个研究型
    Agent，分别调查不同技术方案的可行性，汇总结果辅助决策。

-   开源项目自动化维护：自动处理 GitHub Issue、生成
    PR、执行代码审查，降低项目维护负担。

-   教育与实验平台：作为 AI 编码能力的教学和研究工具，对比不同 Agent
    的表现和效率。

**2.4 核心理念：Feedback Loop 驱动的自动化**

AgentCommander 的核心理念是\"Feedback Loop
驱动的自动化\"（Feedback-Loop-Driven Automation）。这一理念源自对 AI
编码实践的深刻观察：单纯的\"输入需求、输出代码\"模式在简单任务中有效，但在复杂项目中，AI
需要持续的反馈来纠正方向、验证结果、积累经验。Feedback Loop
的核心循环包含四个步骤：读取任务描述（task.json）→ 执行编码任务 →
自测试验证 → 更新进度日志（process.txt）→ 提交代码 →
进入下一个任务。每一次循环都是一次完整的\"计划-执行-检查-行动\"（PDCA）迭代，确保开发质量持续提升。

**3 系统架构设计**

**3.1 整体架构概述**

AgentCommander
采用经典的三层架构设计，各层之间职责清晰、耦合度低，便于独立开发和扩展。三层架构自上而下分别为：用户界面层（UI
Layer）、后端服务层（Backend Service Layer）和智能体执行层（Agent
Execution Layer）。

用户界面层是开发者与平台交互的主要入口，基于 React 构建，提供 Agent
管理、任务看板、实时日志、配置面板等可视化组件。该层通过 RESTful API
与后端服务层通信，通过 WebSocket
获取实时事件推送。后端服务层是平台的核心控制中枢，负责任务调度、Agent
生命周期管理、事件路由、数据持久化等关键业务逻辑。智能体执行层是最底层的执行环境，通过
tmux 会话管理各个 AI Agent 的进程，并通过 Claude Code Hooks 机制捕获
Agent 的行为事件。

**3.2 架构设计原则**

-   关注点分离（Separation of
    Concerns）：每个模块只负责单一职责，模块间通过明确定义的接口通信。UI
    层不直接操作进程，执行层不直接访问数据库。

-   事件驱动（Event-Driven）：模块间通信采用发布-订阅模式，通过
    Communication Bus 传递事件，实现松耦合。

-   可观测性（Observability）：所有 Agent 行为通过 Hooks
    事件采集，全链路日志追踪，支持回溯和审计。

-   容错与恢复（Fault Tolerance）：Agent
    异常退出时自动重启，任务失败时支持重试和回滚，系统崩溃后可从持久化状态恢复。

-   可扩展性（Extensibility）：支持插件化 Agent 类型，新的 AI
    工具只需实现标准接口即可接入平台。

**3.3 关键组件**

Agent Manager（智能体管理器）：负责 Agent
的创建、配置、启动、停止和销毁等全生命周期管理。每个 Agent
拥有独立的配置文件，包含角色定义、系统提示词、工具权限、资源限制等参数。Agent
Manager 还负责维护 Agent 的状态机（idle → starting → working →
needs_input → stopping → terminated），并向上层暴露统一的 Agent
控制接口。

Task
Scheduler（任务调度器）：负责任务的分配、优先级排序和依赖解析。调度器基于
DAG（有向无环图）描述任务间的依赖关系，支持拓扑排序确定执行顺序。在自动模式下，调度器根据
Agent 的空闲状态和能力匹配自动分配任务；在手动模式下，开发者可以手动指定
Agent 执行特定任务。

Process Manager（进程管理器）：负责底层进程（tmux
会话）的创建和监控。通过 node-pty 或 child_process
模块与操作系统交互，管理每个 Agent 的 CLI 进程。Process Manager
还负责资源清理，确保 Agent 终止后相关进程和临时文件被正确释放。

Communication Bus（通信总线）：基于 EventEmitter
实现的进程内消息总线，负责模块间的事件传递。所有 Agent
行为事件（工具调用、文件操作、对话消息等）通过 Communication Bus
分发到各个订阅者（事件日志、WebSocket 推送、告警系统等）。

**4 核心模块设计**

**4.1 Agent 管理模块**

**4.1.1 Agent 定义**

每个 Agent 在 AgentCommander
中通过结构化的配置文件进行定义。配置文件包含以下核心字段：角色标识（role_id）唯一标识
Agent
类型；角色名称（role_name）用于界面展示；系统提示词（system_prompt）定义
Agent 的行为准则和专业知识范围；工具权限（capabilities）指定 Agent
可使用的工具集合（如文件编辑、终端命令、浏览器操作等）；资源限制（resource_limits）设定
Agent 可访问的文件路径、最大 token 预算、最大执行时间等约束条件。

**4.1.2 Agent 生命周期**

Agent 的生命周期包含六个状态：创建（Created）→ 配置（Configured）→
启动（Starting）→ 工作中（Working）→ 等待输入（Needs Input）→
停止（Stopping）→ 已终止（Terminated）。状态转换由 Agent Manager
统一管理，每次状态变更都会产生事件通知，通过 Communication Bus
分发到订阅者。Agent 在启动时，Process Manager 会创建独立的 tmux
会话，并在该会话中启动对应的 AI CLI
工具。配置阶段会将系统提示词、项目上下文和任务描述注入到 Agent
的初始上下文中。

**4.1.3 Agent 类型**

平台预定义以下六种标准 Agent
类型，覆盖软件开发的主要角色。每种类型都经过精心调优的系统提示词和工具权限配置。

**4.1.4 Hook 注入机制**

AgentCommander 通过 Claude Code 的 Hooks 系统实现对 Agent
行为的实时监控。具体实现为：在 Agent 启动时，系统将一个自定义的 Hook
配置文件注入到 Agent 的工作目录中。该配置文件定义了四种类型的 Hook
事件处理器：PreToolUse（工具调用前拦截）、PostToolUse（工具调用后采集）、Notification（通知事件采集）和
Stop（Agent 停止事件采集）。每个 Hook 事件处理器是一个轻量级的 Shell
脚本，通过 HTTP POST
请求将事件数据发送到后端服务的事件收集端点。同时，所有事件也以 JSONL
格式追加写入本地日志文件，确保数据的完整性和可恢复性。

**4.1.5 Agent 通信协议**

Agent 间的通信采用间接消息传递模式，避免直接耦合。通信过程通过
Communication Bus 中转：Agent A
将消息发布到总线，总线根据消息类型和目标路由到 Agent
B。消息格式遵循统一的 JSON
Schema，包含消息类型（message_type）、发送者（sender）、接收者（receiver）、时间戳（timestamp）和消息体（payload）等标准字段。支持的通信场景包括：任务移交（Agent
完成任务后通知下游 Agent 开始）、代码审查请求（Coder 提交代码后自动触发
Reviewer 审查）、冲突通知（文件修改冲突时通知相关
Agent）以及状态广播（Agent 状态变更通知所有关注方）。

**4.2 任务管理模块**

**4.2.1 任务 JSON 结构设计**

每个任务由一个 task.json 文件定义，存储在项目的 .agentcommander/tasks/
目录下。task.json 文件采用标准化的 JSON 结构，包含以下核心字段：任务
ID（id）使用 UUID
全局唯一标识；任务标题（title）简洁描述任务内容；任务描述（description）详细的执行要求和验收标准；任务状态（status）枚举值包含
backlog、in_progress、needs_input、completed、failed；优先级（priority）从
P0（紧急）到 P3（低优）四个等级；依赖列表（dependencies）引用前置任务的
ID 数组；分配的 Agent（assigned_agent）当前执行任务的 Agent
标识；时间戳（timestamps）记录创建、开始、完成等关键时间节点；标签（tags）用于分类和过滤。

**4.2.2 任务生命周期**

任务从创建到完成经历完整的状态机转换：Backlog（待办）→ In
Progress（进行中）→ Needs Input（需要人工输入）→ Completed（已完成）或
Failed（已失败）。在自动模式下，任务状态由系统根据 Agent
的行为自动更新：当 Agent 开始处理任务时，状态从 Backlog 转为 In
Progress；当 Agent 遇到无法自主决策的问题时，状态转为 Needs
Input，系统通知开发者介入；当任务通过自测试验证时，状态转为
Completed。在手动模式下，开发者可以在界面上手动调整任务状态，或通过快捷操作按钮（Start、Done、Cancel）触发状态转换。

**4.2.3 任务依赖图（DAG）**

任务间的依赖关系通过有向无环图（DAG）进行建模和可视化。每个任务节点包含任务
ID、标题和当前状态；每条有向边表示\"前置→后续\"的依赖关系。调度器基于
DAG
进行拓扑排序，确定任务的合法执行顺序。只有当所有前置任务完成后，后续任务才可被分配执行。DAG
还支持并行执行：当多个任务的前置条件同时满足时，它们可以被分配给不同的
Agent
并行处理。平台在前端以交互式流程图的形式展示任务依赖关系，支持拖拽调整依赖和手动重新排序。

**4.2.4 任务优先级系统**

任务优先级采用四级分类体系：P0（紧急阻塞）：阻碍其他任务推进的关键任务，需要立即处理；P1（高优先级）：核心功能开发任务，按计划执行；P2（中优先级）：增强功能和优化任务，可适当延后；P3（低优先级）：非紧急的改进和文档任务。优先级不仅影响手动模式下的任务排序，也是自动调度器分配任务的重要参考因素。调度器在分配任务时会综合考虑优先级、Agent
能力匹配度和当前负载情况。

**4.2.5 从需求文档拆分任务**

平台支持从 PRD、Issue
列表或其他格式的需求文档自动生成任务队列。拆分过程由一个特殊的\"规划
Agent\"（Planner）完成，它接收需求文档作为输入，分析其中的功能点、技术约束和依赖关系，然后生成结构化的
task.json
文件。拆分策略遵循\"单一职责\"和\"可独立验证\"两个原则：每个任务应该是一个独立可完成的工作单元，且具有明确的完成标准和验证方式。开发者可以在任务生成后进行人工审核和调整，确保任务拆分的合理性和完整性。

**4.3 会话与通信模块**

**4.3.1 tmux 会话管理**

每个 Agent 运行在独立的 tmux 会话中，实现进程级别的隔离。会话命名遵循
agent-{agent_id}-{task_id} 的规范格式，便于识别和管理。tmux
会话的创建、附加、分离和销毁由 Process Manager 统一管理。通过
tmux，平台可以实时捕获 Agent 的终端输出、发送命令到 Agent
的标准输入，以及在不中断 Agent 工作的情况下分离和重新附加会话。tmux
还提供了会话持久化能力------即使后端服务重启，tmux 会话中的 Agent
进程仍然继续运行，平台可以通过重新连接会话恢复管理。

**4.3.2 WebSocket 实时事件流**

后端服务通过 Socket.IO
向前端推送实时事件流。事件类型包括：agent.status_change（Agent
状态变更）、agent.activity（Agent
活动心跳）、task.status_change（任务状态变更）、task.output（任务产出通知）、hook.event（原始
Hook 事件）、system.alert（系统告警）和
log.message（日志消息）。前端通过 Socket.IO
客户端订阅这些事件，实时更新界面状态，如任务看板的卡片移动、Agent
状态指示灯的变化、活动日志的滚动更新等。WebSocket
连接支持自动重连和事件回放，确保网络波动不会导致状态丢失。

**4.3.3 Claude Code Hooks 集成**

Claude Code Hooks 是平台获取 Agent 行为数据的核心通道。系统集成了四种
Hook 类型：PreToolUse 在 Agent
调用工具前触发，可用于审批拦截或参数修改；PostToolUse
在工具调用完成后触发，采集工具调用结果和副作用；Notification 在 Agent
发出通知时触发，如需要人工确认的操作；Stop 在 Agent
停止工作时触发，标记任务执行结束。每个 Hook
事件包含丰富的上下文信息：工具名称、输入参数、执行结果、时间戳、会话标识等。后端收到事件后进行标准化处理，存储到事件数据库，并通过
WebSocket 推送到前端。

**4.3.4 JSONL 日志与 HTTP POST 双通道事件收集**

为兼顾数据可靠性和实时性，Hook 事件采用双通道收集策略。主通道为 HTTP
POST：Hook 脚本将事件实时发送到后端的 REST API
端点，由后端进行事件解析、验证和分发。备用通道为 JSONL
本地日志：所有事件同时以 JSONL（JSON
Lines）格式追加写入本地日志文件（.agentcommander/logs/{session_id}.jsonl），每行一个完整的
JSON
对象。当后端服务不可用时，主通道失败，事件数据仍然安全保存在本地日志中。后端恢复运行后，可以通过读取日志文件补录遗漏的事件，确保数据的完整性。

**4.3.5 对话记录持久化**

Agent
的完整对话历史以会话为单位持久化存储。每个会话的对话记录包含用户消息、Agent
回复、工具调用和系统事件。对话记录存储在 .agentcommander/conversations/
目录下，采用 JSONL
格式，便于增量追加和按行读取。前端提供对话记录查看器，支持按时间范围、Agent、任务等维度筛选和检索。对话记录不仅是调试和分析的重要数据源，也是
Agent 上下文注入的基础------当 Agent
开始新任务时，系统可以从相关历史对话中提取关键信息，帮助 Agent
快速理解项目背景和之前的工作成果。

**4.4 并发控制模块**

**4.4.1 资源锁定机制**

多 Agent
并发工作时，文件冲突是最常见的问题。平台实现三级资源锁定机制：文件级锁定（File
Lock）------当 Agent
需要修改某个文件时，先获取该文件的独占锁，修改完成后释放；目录级锁定（Directory
Lock）------当 Agent
需要在某个目录下批量操作时，可获取整个目录的锁，防止其他 Agent
同时修改该目录下的文件；项目级锁定（Project
Lock）------在执行高风险操作（如数据库迁移、大规模重构）时，可获取整个项目的锁，确保操作的原子性。锁采用超时自动释放机制，防止因
Agent 异常退出导致的死锁问题。

**4.4.2 乐观并发控制**

除了显式锁定外，平台还实现了乐观并发控制（Optimistic Concurrency
Control, OCC）作为补充策略。OCC 不阻塞 Agent 的操作，而是在 Agent
提交修改时检查文件是否在读取后被其他 Agent 修改过。检测机制基于文件的
content hash（SHA-256）：Agent 在开始修改前记录文件
hash，修改完成后对比当前文件
hash，如果不一致则说明存在并发修改，系统会提示冲突并要求解决。OCC
适用于大多数低冲突场景，可以最大化并发度。

**4.4.3 冲突检测与解决**

当检测到文件冲突时，平台提供三种解决策略：自动合并（Auto
Merge）------对于简单的非重叠修改（如不同文件的修改），系统自动合并；智能合并（Smart
Merge）------对于同一文件的不同区域修改，尝试基于 diff
的三路合并；人工介入（Manual
Resolve）------对于复杂的冲突（如同一代码块的不同修改），暂停相关
Agent，通知开发者手动解决。所有冲突及其解决结果都会记录在冲突日志中，作为后续优化调度策略的参考数据。

**4.4.4 Agent 隔离策略**

Agent
隔离是并发控制的基础保障。平台采用多重隔离策略：上下文隔离------每个
Agent 的对话上下文独立管理，不会互相污染；文件系统隔离------每个 Agent
可以配置独立的工作目录，限制其可访问的文件范围；Git
分支隔离------每个任务可在独立的 Git 分支上执行，完成后通过 Merge
Request 合并到主分支，避免直接修改共享代码；进程隔离------每个 Agent
运行在独立的 tmux 会话和系统进程中，一个 Agent 崩溃不会影响其他 Agent。

**4.4.5 死锁预防**

死锁是多 Agent
系统中的严重风险。平台通过以下机制预防死锁：锁超时------每个锁都有最大持有时间，超时后自动释放；锁排序------所有资源按照预定义的全局顺序获取锁，打破循环等待条件；死锁检测------后台线程定期检测锁等待图中的环，一旦发现潜在死锁立即干预；最小锁粒度------鼓励
Agent 使用最细粒度的锁（文件级 \> 目录级 \> 项目级），减少锁竞争的概率。

**4.5 监控与监管模块**

**4.5.1 手动监管模式**

手动监管模式面向希望保持完全控制权的开发者。在此模式下，所有关键操作都需要开发者的显式批准：任务分配需要手动确认；Agent
的工具调用（特别是高危操作如文件删除、Git
push）需要审批；任务完成需要验收确认。界面提供清晰的操作审批面板，展示待审批操作的详细信息（操作类型、影响范围、Agent
建议），支持一键批准或拒绝。手动模式适合对代码质量要求极高的项目，或对
AI 行为尚不信任的初期使用阶段。

**4.5.2 自动监管模式**

自动监管模式是 AgentCommander 的核心价值所在。在此模式下，系统实现完整的
Feedback Loop 自动化：任务自动分配------调度器根据 DAG 和 Agent
状态自动分配下一个可执行的任务；执行自动推进------Agent
自主完成任务，无需人工干预；自测试自动验证------任务完成后自动运行测试套件验证结果；进度自动记录------每次任务进展自动写入
process.txt；代码自动提交------验证通过后自动生成规范的
Commit。开发者可以在任何时候从自动模式切换到手动模式，介入特定的决策点。

**4.5.3 Agent 状态监控**

平台实时监控每个 Agent 的运行状态，提供全局状态仪表盘。Agent
状态包括：Idle（空闲，等待任务）、Working（工作中，正在处理任务）、Needs
Input（需要人工输入）、Offline（离线，进程未运行）。仪表盘展示关键指标：活跃
Agent
数量、当前执行任务数、待处理任务队列长度、平均任务完成时间、资源使用率等。状态变化通过颜色编码直观展示（绿色=正常、黄色=需要关注、红色=异常），并支持按项目和时间段筛选。

**4.5.4 实时活动流**

活动流（Activity
Feed）是展示系统实时动态的核心界面元素，以时间线形式展示所有 Agent
和任务的活动事件。活动事件包括：Agent
状态变化、工具调用记录、任务状态更新、代码提交记录、冲突告警等。每条活动记录包含时间戳、Agent
标识、事件类型和简要描述，点击可展开查看详细信息。活动流支持过滤和搜索，开发者可以快速定位感兴趣的事件。

**4.5.5 告警与通知系统**

平台支持多级别的告警通知机制。告警级别分为
Info（信息通知）、Warning（警告）、Error（错误）和
Critical（严重错误）。告警触发条件包括：Agent
执行超时、任务连续失败超过阈值、文件冲突频率异常、资源使用率超过阈值等。告警通知渠道支持：界面内通知气泡、浏览器原生通知、邮件通知（可配置
SMTP）和 Webhook 回调（可集成 Slack、Discord
等）。开发者可以自定义告警规则和阈值，平衡通知频率与信息完整性。

**5 自动化工作流设计**

**5.1 需求拆分工作流**

**5.1.1 输入**

需求拆分工作流的输入可以是多种格式的需求文档，包括：结构化的 PRD
文档（如本文件）、GitHub Issue 列表、用户故事（User
Stories）、功能需求清单（Feature
List），甚至是一段自然语言描述的需求。系统对输入格式不做强制要求，而是通过
AI 理解能力自动提取需求中的功能点、技术约束和验收标准。

**5.1.2 处理**

处理阶段由 Planner Agent
负责，执行以下步骤：第一步，需求解析------阅读并理解需求文档，识别其中的功能模块和技术依赖；第二步，任务分解------将每个功能模块进一步拆分为可独立执行的任务，遵循\"单一职责\"和\"可独立验证\"原则；第三步，依赖分析------分析任务间的依赖关系，构建
DAG；第四步，优先级评估------根据依赖关系和功能重要性确定任务优先级；第五步，生成
task.json------为每个任务创建标准化的 task.json 文件，写入
.agentcommander/tasks/ 目录。

**5.1.3 输出**

工作流输出为结构化的任务队列，存储在项目的 .agentcommander/tasks/
目录下。每个任务对应一个独立的 task.json
文件，包含完整的任务定义。前端任务看板自动加载这些任务文件，以卡片形式展示任务标题、描述、状态、优先级和依赖关系。开发者可以在看板上审核和调整任务，确认无误后启动自动执行流程。

**5.2 自动执行循环（Feedback Loop）**

**5.2.1 循环步骤**

Feedback Loop 是自动模式下的核心执行机制，每一次循环包含以下步骤：

1.  读取任务描述：Agent 从 task.json
    中读取当前分配的任务，理解任务目标和验收标准。

2.  执行编码任务：Agent 根据任务要求进行代码编写、文件创建或修改等操作。

3.  自测试验证：Agent 运行相关测试用例，验证任务实现是否满足验收标准。

4.  更新进度日志：Agent 将执行结果、遇到的问题和下一步计划写入
    process.txt。

5.  提交代码：验证通过后，Agent 执行 git add 和 git
    commit，生成规范的提交信息。

6.  进入下一个任务：系统将任务状态标记为
    Completed，调度器分配下一个可执行的任务。

**5.2.2 上下文注入方法**

上下文注入是确保 Agent
拥有足够信息完成任务的关键机制。注入的内容包括：process.txt------累积的进度日志，记录所有已完成任务的关键信息和决策历史，Agent
通过阅读 process.txt
可以了解项目当前状态；worklog.md------详细的工作日志，包含每次任务执行的具体操作记录；相关源代码------系统根据任务类型自动提取相关文件的摘要或关键代码段；API
文档------当任务涉及 API 开发时，注入相关的 API 规格说明；git diff
摘要------最近的代码变更摘要，帮助 Agent 理解最新的项目变化。

**5.2.3 自测试协议**

自测试是 Feedback Loop
中确保质量的关键环节。平台设计了三层测试协议：第一层，后端 API
测试------Agent
编写并通过单元测试和集成测试验证后端逻辑的正确性，使用项目配置的测试框架（Jest、pytest
等）执行；第二层，终端验证------Agent 通过 tmux
会话执行启动命令，验证服务是否正常启动、API
是否正确响应；第三层，前端视觉测试------通过浏览器自动化工具（Playwright/Puppeteer）启动前端应用，截图并与预期结果对比，验证
UI 的正确性。只有当所有测试层都通过时，任务才被标记为已完成。

**5.2.4 错误处理与重试策略**

当任务执行失败或自测试未通过时，系统采取以下策略：首先，Agent
自行诊断问题------分析错误日志、测试输出和代码变更，尝试定位和修复问题；其次，有限重试------Agent
最多进行 3
次自动修复尝试，每次尝试都重新运行完整的自测试；再次，状态回退------如果
3 次尝试均失败，Agent 撤销所有代码变更（git reset），将任务状态标记为
Failed，并详细记录失败原因；最后，人工介入------系统通知开发者审查失败任务，开发者可以调整任务描述、提供额外指导或手动解决问题后重新将任务标记为
In Progress。

**5.3 上下文管理**

**5.3.1 上下文隔离的重要性**

AI 编码智能体的一个核心限制是上下文窗口（Context
Window）的有限性。当上下文窗口被过多无关信息填满时，Agent
的注意力会被分散，导致指令遵循度下降、代码质量降低。因此，上下文隔离是
AgentCommander
架构设计中的核心考量。每个任务都应该在新启动的上下文中执行，避免前序任务的长对话历史对当前任务产生干扰。上下文隔离的另一个好处是可恢复性------如果某个任务的执行出现严重偏差，可以直接终止并重新开始，而不影响其他任务的进度。

**5.3.2 新上下文策略**

平台推荐\"新上下文 + 精准注入\"策略：每次任务开始时，Agent
启动一个全新的对话上下文，而不是在之前的对话基础上继续。新上下文通过精心设计的系统提示词进行初始化，注入当前任务所需的关键信息。这种方式确保
Agent
的注意力集中在当前任务上，同时通过注入机制弥补了上下文隔离可能造成的信息缺失。实测表明，新上下文策略虽然每次需要重新建立对项目的理解，但由于注入的信息是经过筛选和压缩的精华内容，整体效率反而优于长对话的累积模式。

**5.3.3 有效上下文注入**

有效上下文注入的核心是在有限的 token
预算内提供最有价值的信息。平台采用以下策略确保注入内容的高效性：process.txt
累积日志------只包含每个已完成任务的一句话摘要和关键决策点，而非完整的执行记录；相关源文件------根据任务描述自动匹配最相关的
3-5 个源代码文件，而非整个项目代码库；API
文档片段------只注入与当前任务直接相关的 API 端点文档；git diff
摘要------压缩展示最近的代码变更，突出修改的文件和函数签名。所有注入内容都经过
token 预算控制，确保不会超出 Agent 的上下文窗口限制。

**5.3.4 记忆管理**

长期记忆管理采用滑动窗口和重要性评分相结合的策略。滑动窗口保留最近 N
个任务的详细信息，超出窗口的任务只保留摘要。重要性评分基于任务的特征自动计算：高优先级任务、涉及核心架构的任务、修复关键
Bug
的任务会获得更高的评分，其详细信息被保留更长的时间。记忆内容按重要性排序存储，当总
token 量超过预算时，优先丢弃重要性评分低的内容。这种机制确保 Agent
始终能访问最重要的项目信息，同时控制上下文的使用量。

**5.4 Git 工作流集成**

**5.4.1 自动提交**

任务完成后，Agent 自动执行 Git
提交操作。自动提交遵循以下规则：只提交与当前任务相关的文件变更，避免将临时文件、配置文件等无关变更混入提交；提交前执行
git status 和 git diff
预览，确认变更范围正确；提交信息遵循规范化格式，包含任务
ID、类型标签和简要描述。自动提交机制通过 Claude Code 的 Stop Hook
触发，确保在 Agent 完成工作后立即提交，减少人为遗漏的风险。

**5.4.2 提交信息规范**

提交信息遵循 Conventional Commits 规范，格式为：{type}({scope}):
{description} \[{task_id}\]。其中 type 可以是
feat（新功能）、fix（修复）、refactor（重构）、test（测试）、docs（文档）、style（样式）、chore（杂项）等；scope
标识影响的模块或组件；description 是简洁的变更描述；task_id 关联到对应的
task.json。例如：feat(auth): implement JWT token refresh
\[task-0042\]。规范的提交信息不仅便于代码审查，也为后续的变更日志（Changelog）自动生成提供了数据基础。

**5.4.3 分支策略**

平台支持两种分支策略，可根据项目需求灵活选择：顺序执行模式------所有任务在主分支（main/develop）上顺序执行，适用于小型项目和快速原型开发，优点是流程简单、无合并冲突；任务分支模式------每个任务在独立的
feature 分支上执行，完成后创建 Pull Request
合并回主分支，适用于大型项目和团队协作场景，优点是隔离性好、支持并行开发和代码审查。分支策略可以在项目配置中指定，也可以在运行时通过界面切换。

**5.4.4 PR 生成**

在任务分支模式下，任务完成后平台自动生成 Pull Request。PR 标题和描述由
Agent
根据任务描述和实际代码变更自动生成，包含变更摘要、影响范围和测试结果。PR
标签根据任务类型自动设置（如
feature、bugfix、refactor）。对于需要人工审查的
PR，系统通过通知渠道提醒相关开发者。未来版本计划集成 AI
代码审查能力，自动在 PR 中添加审查意见，进一步降低人工审查负担。

**6.2.1 三面板主布局**

主界面采用经典的三面板布局设计，最大化信息密度和操作效率。左侧面板（Agent

Agent 配置面板支持创建和编辑 Agent

token

Agent 的系统提示词完整效果。支持从已有 Agent

**7.2 API 设计概览**

**7.2.1 RESTful API 端点**

后端 API 遵循 RESTful
设计规范，按资源组织端点。核心端点包括：项目管理------GET/POST
/api/projects、GET/PUT/DELETE /api/projects/:id；Agent
管理------GET/POST /api/agents、GET/PUT/DELETE /api/agents/:id、POST
/api/agents/:id/start、POST /api/agents/:id/stop；任务管理------GET/POST
/api/tasks、GET/PUT/DELETE /api/tasks/:id、PATCH
/api/tasks/:id/status；事件查询------GET
/api/events（支持分页、过滤和排序）；项目管理------GET/POST
/api/projects/:id/settings。API 统一使用 JSON 格式，错误响应遵循 RFC
7807（Problem Details for HTTP APIs）规范。

**7.2.2 WebSocket 事件**

WebSocket
事件是实时通信的核心通道。服务端推送的事件类型包括：agent:status_update（Agent
状态变更，含
agent_id、old_status、new_status、timestamp）；agent:activity（Agent
活动心跳，含当前操作描述和持续时间）；task:status_update（任务状态变更）；task:output（任务产出通知，含产出类型和内容摘要）；hook:event（原始
Hook
事件转发，含完整事件数据）；system:alert（系统告警，含告警级别和详细信息）；log:entry（日志条目推送）。客户端可以发送的事件包括：agent:command（向
Agent
发送命令）、task:action（任务操作指令）、subscription:update（订阅/取消订阅特定事件类型）。

**7.3 进程管理**

**7.3.1 tmux 集成策略**

后端通过 node-pty 库与 tmux 进行集成，实现以下能力：创建 tmux
会话------使用 tmux new-session -d -s {name}
命令创建后台会话；向会话发送命令------使用 tmux send-keys -t {session}
\'{command}\' Enter 发送命令；捕获会话输出------使用 tmux capture-pane
-t {session} -p 捕获当前面板内容；销毁会话------使用 tmux kill-session
-t {name} 清理会话。所有 tmux 操作都封装在 ProcessManager
类中，对外提供高层的 startAgent、sendCommand、getOutput、stopAgent
等方法。

**7.3.2 进程生命周期管理**

进程管理器维护一个进程注册表（Process Registry），记录所有活跃的 Agent
进程信息，包括 PID、tmux
会话名、启动时间、资源使用量等。进程管理器通过定时心跳检测（每 5
秒）监控进程健康状态。当检测到进程异常退出时，管理器触发以下流程：记录异常退出事件
→ 尝试获取进程的退出码和错误输出 → 根据配置决定是否自动重启 → 更新 Agent
状态为 Offline → 通知前端和告警系统。

**7.3.3 资源清理与垃圾回收**

系统在以下场景执行资源清理：Agent 正常停止时------终止 tmux
会话、释放文件锁、清理临时文件；Agent
异常退出时------强制终止关联进程、释放锁、保存最后的终端输出到日志；服务关闭时------优雅停止所有
Agent
进程、保存系统状态、持久化未完成任务的状态。垃圾回收器定期扫描并清理以下资源：超时的文件锁（超过配置的最大持有时间）、过期的临时文件（超过
24 小时）、过期的日志文件（根据日志保留策略配置）和孤立进程（没有对应
Agent 记录但仍在运行的 tmux 会话）。

**8.3 实体关系描述**

数据实体间的关系如下：Project 与 Agent
是一对多关系------一个项目可以包含多个 Agent，每个 Agent
只属于一个项目。Project 与 Task
也是一对多关系------一个项目包含多个任务。Agent 与 Task
是一对多关系------一个 Agent
可以被分配多个任务（顺序执行），但一个任务同一时间只分配给一个
Agent。Task 与 Task（通过 task_dependencies
表）是多对多关系------一个任务可以依赖多个前置任务，也可以被多个后续任务依赖。Event
与 Agent/Task 是多对一关系------事件由特定 Agent
在执行特定任务时产生。Conversation 与 Agent/Task
同样是多对一关系------对话记录关联到产生对话的 Agent 和上下文任务。

**9 开发路线图**

AgentCommander
的开发采用渐进式迭代策略，分四个阶段推进。每个阶段都有明确的目标和交付物，确保项目在每个里程碑都能交付可用的功能增量。

**9.1 Phase 1：MVP（第 1-6 周）**

MVP 阶段的目标是验证核心概念并交付最小可用产品。核心功能包括：Agent
的基本管理（创建、配置、启动、停止）、任务的 CRUD 操作和看板展示、tmux
会话管理和终端输出查看、基本的 WebSocket 实时事件推送、SQLite
数据库集成以及简洁的深色主题
UI。此阶段的交付标准是：开发者可以通过界面创建 Agent、分配任务、查看
Agent 的实时输出，并手动管理任务状态。

**9.2 Phase 2：自动化（第 7-10 周）**

自动化阶段是产品价值的核心体现。在此阶段将实现：自动监管模式和 Feedback
Loop 执行引擎、Claude Code Hooks
集成和双通道事件收集、任务依赖图（DAG）和自动调度、需求文档到任务队列的自动拆分、自测试协议（后端
API 测试 +
终端验证）以及上下文注入和记忆管理机制。此阶段的交付标准是：给定一份 PRD
文档，系统能够自动拆解为任务队列，并调度 Agent 逐步完成开发和自测试。

**9.3 Phase 3：打磨（第 11-13 周）**

打磨阶段聚焦于用户体验和系统稳定性。主要工作包括：高级监控仪表盘（任务进度图表、Agent
负载分析）、前端视觉测试（浏览器自动化 +
截图对比）、告警和通知系统完善、UI
交互优化（拖拽操作、动画过渡）、错误处理和边缘场景覆盖、性能优化（大量事件下的界面流畅度）以及多项目管理支持。此阶段的交付标准是：产品在功能和体验上达到可以公开展示的质量水平。

**9.4 Phase 4：规模化（第 14-19 周）**

规模化阶段为产品的长期发展奠定基础。核心工作包括：多用户支持和权限管理、PostgreSQL
数据库迁移和团队协作功能、云端部署方案（Docker + 云服务）、Agent
插件市场基础架构、API
开放平台（允许第三方集成）以及详细的使用文档和最佳实践指南。此阶段的交付标准是：产品可以作为
SaaS 服务面向团队用户开放使用。

**10 风险分析与应对**

**10.1 技术风险**

-   Agent 行为不可预测性：AI
    智能体可能产生偏离预期的行为，如修改错误的文件、产生有 bug
    的代码或陷入无限循环。应对策略：通过 Hooks 机制实时监控 Agent
    行为，设置文件访问白名单和操作权限限制，实现操作审批机制（手动模式下）和自动回退机制（自动模式下），以及设置最大执行时间和最大
    token 消耗限制。

-   API 速率限制和配额消耗：AI 编码工具的 API
    调用可能受到速率限制或产生高昂的费用。应对策略：在 Agent 配置中设置
    token 预算上限，实现请求排队和速率控制，提供 API
    使用量监控仪表盘，支持多个 API Key
    的负载均衡轮换，以及优先级调度------确保高优先级任务的 API
    配额不被低优先级任务耗尽。

-   tmux 会话管理复杂性：tmux
    会话的创建、监控和清理在不同操作系统上可能表现不一致。应对策略：封装所有
    tmux
    操作为统一接口，添加全面的错误处理和重试机制，实现会话健康检查和自动恢复，以及提供
    fallback 模式（不使用 tmux，直接使用 child_process）。

**10.2 架构风险**

-   可扩展性瓶颈：当 Agent
    和任务数量增长时，单进程架构可能成为性能瓶颈。应对策略：后端架构从一开始就考虑水平扩展能力，使用事件驱动架构便于未来拆分为微服务，数据库访问层采用抽象接口便于切换，以及关键路径（事件处理、日志写入）使用异步非阻塞操作。

-   容错能力不足：系统崩溃可能导致任务状态丢失或 Agent
    进程成为孤儿进程。应对策略：所有关键状态变更实时持久化到数据库，实现优雅关闭（Graceful
    Shutdown）流程，启动时执行系统自检和孤儿进程清理，以及设计状态恢复机制（从数据库恢复崩溃前的运行状态）。

**10.3 用户体验风险**

-   学习曲线陡峭：多 Agent
    编排是一个新概念，用户可能需要较长时间理解和使用。应对策略：提供交互式的新手引导（Onboarding
    Wizard），内置预配置的项目模板（如"全栈 Web 应用"、"API
    服务"），提供丰富的示例和教程文档，以及设计直观的界面隐喻（如将
    Agent 类比为团队成员、将任务看板类比于项目管理工具）。

-   配置复杂度高：Agent
    的系统提示词、工具权限和资源限制等配置项可能让用户感到困惑。应对策略：提供智能配置向导，根据任务类型自动推荐配置，内置经过优化的预设模板，以及支持配置导入导出和社区分享。

**10.4 应对策略总结**

风险管理的核心原则是\"渐进式暴露\"和\"安全网兜底\"。渐进式暴露意味着系统默认使用安全保守的配置（如手动监管模式、严格的文件权限），用户可以随着经验的积累逐步开启更高级的自动化功能。安全网兜底意味着即使出现最坏的情况（Agent
完全失控），系统也有最终的安全措施（进程强制终止、代码变更回滚、状态快照恢复）确保不会造成不可逆的损失。这种设计理念让用户在使用过程中始终保持掌控感，是产品获得用户信任的基础。

**11 参考项目与灵感**

AgentCommander
的设计和实现借鉴了多个优秀的开源项目和产品理念，以下是主要的参考来源和各自的启发点。

**11.1 AgentKing**

AgentKing 是一个多 AI 智能体管理平台的参考实现，其 UI/UX 设计为
AgentCommander 提供了重要的视觉参考。从 AgentKing
中借鉴的设计理念包括：三面板布局（Agent 列表 + 任务区域 +
详情面板）的信息架构、深色主题作为默认视觉风格、Agent
状态的视觉编码（颜色指示灯 +
状态文字）以及任务卡片的紧凑信息展示方式。AgentKing
证明了此类工具的界面设计可行性，也帮助识别了一些可以改进的交互痛点。

**11.2 auto-coding-agent-demo**

auto-coding-agent-demo 是一个 10 小时 AI 全自动编码实验项目，展示了 AI
智能体从零开始自主完成编码项目的完整过程。该项目为 AgentCommander
的自动执行模式提供了核心灵感：基于文件系统的任务管理（task.json
定义任务），Feedback Loop 执行模式（执行 → 自测 → 提交 →
下一任务），以及 process.txt
进度日志作为上下文传递媒介。auto-coding-agent-demo
证明了全自动编码在技术上的可行性，而 AgentCommander
将这一理念系统化、平台化，使其适用于更广泛的项目类型和开发场景。

**11.3 planning-with-files**

planning-with-files 是一个 Claude Code Skill，实现了 Manus
风格的基于文件的持久化规划模式。该项目的核心理念------通过文件（而非对话历史）来保持和传递上下文------深刻影响了
AgentCommander 的上下文管理设计。具体借鉴包括：process.txt
作为累积进度日志的设计模式、worklog.md
作为详细工作记录的实践、以及\"新上下文 +
文件注入\"优于\"长对话累积\"的上下文管理策略。planning-with-files
证明了文件系统的持久化能力是 AI 编码工具的重要基础设施。

**11.4 Claude Code Hooks 系统**

Claude Code 提供的 Hooks 系统是 AgentCommander 实现 Agent
行为监控的核心技术基础。Hooks 系统支持在 AI
工具的关键行为节点（工具调用前、工具调用后、通知、停止）注入自定义逻辑，使得外部系统可以实时感知和干预
AI 的行为。AgentCommander
充分利用了这一能力，构建了双通道事件收集系统，实现了对 Agent
行为的全链路监控。Hooks
系统的设计哲学------可编程的、事件驱动的工具行为扩展机制------也启发了
AgentCommander 自身的插件化架构设计。

**11.5 Manus AI Agent 范式**

Manus AI Agent 是一个通用的 AI Agent 平台，其设计范式对 AgentCommander
产生了深远影响。Manus 的核心理念包括：Agent
应该拥有持久的记忆和规划能力、Agent 的工作应该可以被人类监督和干预、以及
Agent 的执行过程应该是可观测和可解释的。这些理念与 AgentCommander
的设计目标高度一致，尤其在监控与监管模块的设计中得到了充分体现。Manus
还展示了\"AI 助手 + 人类监督\"的协作模式是当前 AI Agent
产品最实用的形态。
