CHAPTER 01

01 产品概述

定义系统的背景、定位、核心概念与关键术语。

1.1 项目背景

随着大语言模型（LLM）和 AI Agent 技术的快速发展，利用智能化代理完成复杂软件工程任务已成为行业趋势。Claude Code CLI 等工具的出现，使得 AI Agent 可以直接在本地终端中执行代码编写、调试、测试等操作，极大地拓展了自动化的边界。然而，当前大多数 Agent 系统存在以下核心痛点：单个 Agent 承担过多职责导致上下文过载，难以处理大型复杂任务；Agent 之间缺乏有效的协同机制，无法实现真正的并行化执行；系统的状态管理依赖进程内存，一旦崩溃便丢失全部进度；缺乏完善的可观测性手段，任务执行过程如同黑箱，难以进行有效的调试和监控；用户无法直观地管理和监控多个 Agent 的实时工作状态。

为了解决上述问题，本项目提出了一种基于 Web 应用 + 主从架构的分布式 Agent 自动化执行系统。该系统借鉴了 Ralph Shell 循环执行的核心思想，将其升级为多 Agent 协同模式。用户通过 Web 应用管理面板作为统一入口，系统默认启动一个 Master Agent（Claude Code CLI），扮演编排者/老板的角色，并通过 tmux 终端复用器动态创建和管理多个 Sub-Agent CLI 实例。所有状态通过文件系统持久化，结合 BDD 需求精炼、TDD 质量保障、Git 版本管理、OpenClaw 进程监控与移动端告警机制，打造一个端到端的高可用、可扩展、可观测的自动化任务执行平台。

1.2 产品定位

本系统的核心定位是：一个以 Web 应用为入口、以 Claude Code CLI 为执行引擎、以 tmux 为终端管理中枢的多 Agent 分布式协同平台。系统的使用模式可以概括为以下场景：用户打开 Web 应用管理面板，在 UI 上看到各个 Agent 的实时工作状态。系统默认先开启一个 Master Agent Claude Code CLI，并赋予其编排者/老板的角色特征以及 Harness 约束工程规范。用户通过面板或终端给老板下达真正要实现的需求，老板则通过 tmux 继续增加新的 CLI Sub-Agent 实例来完成不同阶段的工作。

1.3 核心概念

本系统引入了"角色化 Agent"的核心概念。每个 Claude Code CLI 实例在启动时都会被赋予特定的角色特征和 Harness 约束，例如产品经理、高级开发工程师、高级编剧、测试工程师等。角色化设计确保每个 Agent 都在明确的职责边界内工作，不会越权执行不属于自身职责的任务。Master Agent 作为编排者/老板，负责全局决策和任务调度，但不执行具体的业务逻辑。Sub-Agent 在接收到任务后独立执行，完成后更新进度并通过 Git 提交代码，然后自动退出，实现真正的无状态执行。

系统的工作规模可以非常大——Product Manager Agent 拆解出的 task.json 中可能包含几十甚至上百个原子任务，Master Agent 需要智能决策哪些任务需要哪些类型的 Sub-Agent 来完成，并根据任务间的依赖关系合理安排执行顺序和并行策略。

CHAPTER 02

02 设计原则

五大核心原则：Web 优先入口、角色化执行、无状态通信、全链路可观测、工程化闭环。

2.1 Web 优先入口原则

用户交互层统一通过 Web 应用管理面板进入，而非直接操作终端。Web 面板提供直观的 UI 来查看和管理各个 Agent 在做什么，包括实时状态、任务进度、日志输出、终端连接等。底层通过 tmux 连接本地的 Claude Code CLI 等工具进行实际工作，用户无需关心底层终端细节，也可以随时通过面板切换到某个 Agent 的终端进行直接交互。这种"Web 入口 + 终端执行"的分层设计，既降低了使用门槛，又保留了终端操作的灵活性。

2.2 角色化执行原则

每个 Claude Code CLI 实例在启动时被赋予明确的角色特征和 Harness 约束。Master Agent 是编排者/老板，负责全局决策和任务调度，不执行具体业务逻辑。Product Manager Agent 负责需求精炼和任务拆解。高级开发工程师 Agent 负责编码实现。测试工程师 Agent 负责编写和执行测试脚本。角色化的好处在于：每个 Agent 都清楚自己的职责边界，不会越权操作；Harness 约束确保行为规范一致；角色可以灵活扩展，添加新的角色类型无需修改核心调度逻辑。

2.3 无状态通信原则

所有 Sub-Agent 在执行过程中不保留任何上下文记忆，每个 Sub-Agent 的输入参数、执行过程状态、输出结果均通过文件系统进行持久化存储和传递。这种设计确保了：任何 Agent 进程崩溃都不影响系统整体状态；Master Agent 可以随时启动新的 Sub-Agent 进程处理任务，无需考虑状态恢复；文件系统天然支持并发读写，多个 Sub-Agent 可以安全地同时操作不同的文件，实现真正的并行执行。每个 Sub-Agent 的完整生命周期为：启动 → 读取输入 → 执行任务 → 写入输出 → Git 提交 → 更新进度 → 退出进程。

2.4 全链路可观测原则

系统从设计之初就将可观测性作为核心功能。三层可观测机制覆盖了不同维度：文件层通过 progress.md 提供全局任务视图，memory.md 记录知识累积，dynamic_log.md 记录实时操作日志；进程层通过 tmux 提供每个 Agent 的实时终端输出查看；监控层通过 OpenClaw 进行最底层的进程存活检测、超时检测和资源监控。此外，Web 管理面板将所有可观测数据汇总展示，用户可以一站式查看系统全局状态。

2.5 工程化闭环原则

需求定义阶段采用 BDD 格式，Product Manager Agent 通过 Given-When-Then 结构与用户反复沟通，确保每个功能细节都被精确定义，杜绝需求模糊。每个 task 都配有对应的测试脚本（可以使用 Playwright CLI 进行浏览器操作验证和截图确认等）。开发阶段采用 TDD 流程，先编写测试再编写功能代码。每个 task 完成并通过测试后，自动进行 Git 提交，将本任务所做的所有代码变更提交到版本控制系统，确保开发过程完全可追溯。BDD + TDD + Git 三者形成了一个完整的工程化闭环。

3.3 组件间通信机制

本系统采用"文件系统 + tmux"双通道通信模式。文件系统通道负责持久化数据的异步传递：Master Agent 写入 task.json 定义任务，Sub-Agent 从中读取任务描述；Sub-Agent 完成后将结果写入 progress.txt 和 memory.md，其他组件从中读取最新状态；所有组件将操作日志追加到 logs/ 目录下的日志文件。tmux 通道负责实时交互：Web 面板通过 tmux 连接到特定 Agent 的终端会话，实现实时输出查看和指令输入；Master Agent 通过 tmux send-keys 命令向 Sub-Agent 发送指令。两个通道互为补充，文件系统保证可靠性和持久性，tmux 保证实时性和交互性。

CHAPTER 04

04 核心功能详细说明

Web 管理面板、Claude Code CLI 集成、Master Agent 调度引擎、Sub-Agent 执行框架、tmux 终端管理、OpenClaw 监控告警、Git 版本管理。

4.1 Web 应用管理面板

4.1.1 核心功能

Web 应用管理面板是用户与整个系统交互的唯一入口。面板提供以下核心功能：Agent 状态总览，以卡片或列表形式展示当前所有 Agent（Master Agent 和各 Sub-Agent）的实时状态，包括运行中、等待中、已完成、失败等状态标识；任务进度看板，以看板或甘特图形式展示 task.json 中所有任务的执行进度和依赖关系；终端视图，嵌入式的终端窗口，用户可以通过面板直接查看任意 Agent 的 tmux 终端输出，也可以切换到交互模式发送指令；需求交互界面，用于 Product Manager Agent 与用户之间进行 BDD 需求确认的交互面板；告警通知中心，展示 OpenClaw 推送的历史告警记录和当前待处理的人工介入请求。

4.1.2 技术架构

Web 应用采用前后端分离架构。前端使用 React 或 Vue 构建 SPA 应用，通过 WebSocket 与后端保持实时连接，获取 Agent 状态更新和日志流。后端负责与 tmux Server 交互（通过 libtmux 或 tmux 命令行），读取文件系统状态数据，转发用户指令。Web 应用默认监听本地端口，用户在浏览器中访问即可使用。在安全方面，Web 应用仅监听 localhost，不暴露到公网；如需远程访问，建议通过 SSH 隧道或 VPN。

4.2 Claude Code CLI 集成

Claude Code CLI 是本系统中每个 Agent 实例的执行引擎。系统通过 tmux 创建新的终端窗口，并在其中启动 Claude Code CLI 进程。每个 CLI 实例启动时，Master Agent 会通过 tmux send-keys 将角色定义和 Harness 约束注入到 CLI 中，使其具备特定角色的行为特征。例如，启动一个"高级开发工程师"Sub-Agent 时，会注入 sub-rules.md 中定义的职责范围、编码规范、提交规则等约束。Claude Code CLI 具备强大的代码理解、生成和调试能力，结合角色化 Harness 约束，可以高效地完成各类型任务。tmux 的会话持久化特性确保即使 SSH 断开，Agent 进程仍在后台继续执行。

4.3 主调度 Agent（Master Agent）

4.3.1 角色定位

Master Agent 是系统的编排者/老板，由一个默认启动的 Claude Code CLI 实例承载。它被赋予了 main-rules.md Harness 约束，明确其职责范围包括：接收和解读用户的原始需求；通过 tmux 创建和管理 Sub-Agent CLI 实例；决策任务拆解策略和分配方案；监控全局执行进度；处理任务失败和异常恢复。Master Agent 不执行具体的业务逻辑——它不编写代码、不运行测试、不做数据分析，这些工作全部委派给具有对应角色特质的 Sub-Agent 完成。

4.5 终端管理层（tmux 集成）

tmux 作为终端管理层是整个系统的核心基础设施之一。Master Agent 通过 tmux 实现所有 Agent CLI 实例的创建、管理和通信。具体集成方式：Master Agent 使用 tmux new-session 创建主会话，使用 tmux new-window 为每个 Sub-Agent 创建独立的终端窗口。窗口命名采用约定格式 agent_{角色}_{序号}，例如 agent_pm_1 表示第一个产品经理 Agent，agent_dev_3 表示第三个开发工程师 Agent。通过 tmux send-keys 命令可以向指定窗口发送文本指令，实现 Master Agent 与 Sub-Agent 之间的实时通信。tmux 的会话持久化特性确保即使 SSH 连接断开或 Web 面板关闭，所有 Agent 进程仍在后台继续运行。Web 管理面板通过集成 tmux 终端视图，让用户可以实时查看任意 Agent 的终端输出。

4.6 进程监控与告警层（OpenClaw 集成）

OpenClaw 是独立于 Agent 执行系统的外部监控进程，作为最后一道安全防线存在。它的独立性至关重要——即使 Master Agent 自身出现异常崩溃，OpenClaw 仍能正常工作并发送告警通知。OpenClaw 的核心监控能力包括三个方面：进程存活检测，定期检查所有 Claude Code CLI 进程是否仍在运行；执行超时检测，对每个 task 设定最大执行时间，超时则标记为异常；资源使用监控，监控 CPU、内存、磁盘等资源使用情况，防止 Agent 进程失控占用过多系统资源。检测到异常时，OpenClaw 通过移动端推送渠道（企业微信、钉钉、Slack）发送告警通知，通知用户需要人工介入。同时 OpenClaw 支持 Master Agent 的反向通知：当 Master Agent 发现某个 task 执行失败时，会主动通知 OpenClaw 触发告警并暂停整个任务流，等待人工给出修复方案。

4.7 Git 版本管理集成

Git 版本管理是本系统工程化闭环的关键环节。每个 Sub-Agent 在成功完成一个 task 且测试通过后，必须将本任务所做的所有代码变更通过 git commit 提交到版本控制系统。提交信息遵循约定格式：[task-{id}] {角色}: {简要描述}，例如 [task-023] senior-developer: 实现用户登录页面布局和样式。如果任务执行失败，Sub-Agent 也会记录失败信息并更新 task.json 中的状态，但不会进行 git commit。Master Agent 在整个任务流完成后，会生成一份完整的执行报告，包含每个 task 的执行状态、git commit hash、执行时间等元数据。这种设计确保了整个开发过程的每一步都可追溯到具体的 commit，极大地方便了代码审查和问题排查。

CHAPTER 05

05 端到端工作流

从用户输入需求到全部任务完成，三大阶段：需求定义、任务执行、失败恢复。

5.1 第一阶段：需求定义（BDD 精炼）

需求定义是整个工作流的起点，也是决定项目成败的关键阶段。当用户通过 Web 管理面板下达原始需求后，Master Agent 会作为编排者/老板，通过 tmux 创建一个"产品经理"Sub-Agent（Claude Code CLI 实例），并将原始需求传递给它。Product Manager Agent 的核心职责是将模糊的大需求拆解为精确的原子任务列表，并通过 BDD 方式与用户反复沟通，确保每个功能细节都被准确定义，彻底杜绝需求模糊。

5.1.1 BDD 需求精炼流程

Product Manager Agent 采用行为驱动开发（BDD）方法与用户进行结构化沟通。对于每个功能需求，PM Agent 会使用 Given-When-Then 格式描述预期的行为：

  • Given（前置条件）：描述系统当前状态，例如"用户已登录且在首页"
  • When（操作触发）：描述用户或系统的操作，例如"用户点击搜索按钮并输入关键词"
  • Then（预期结果）：描述系统的响应，例如"展示搜索结果列表，每条结果包含标题、摘要和链接"
  • 沟通迭代：PM Agent 将 BDD 描述展示给用户确认，用户可以提出修改意见，PM Agent 据此调整并重新描述，直到用户完全确认无误
  • 任务拆解：确认后的每个功能被拆解为一个或多个原子 task，每个 task 有明确的输入、输出、验收标准

5.1.2 测试脚本编写

需求精炼完成后，PM Agent 还需要为每个 task 编写对应的测试脚本。测试脚本的类型取决于任务的性质：对于前端 UI 任务，可以使用 Playwright CLI 进行浏览器自动化测试，包括页面元素验证、交互操作模拟、截图对比等；对于 API 接口任务，编写 HTTP 请求测试脚本验证接口的请求参数、响应格式和状态码；对于数据处理任务，编写输入输出对比测试确保数据转换的正确性。测试脚本和 BDD 描述一起构成 task.json 的完整内容，确保每个 task 都有清晰的"怎么做"和"怎么验证"。

5.1.3 task.json 交付

当所有 task 的 BDD 描述和测试脚本都经过用户确认后，PM Agent 将完整的 task.json 交付给 Master Agent。一个典型的中型项目可能包含几十个 task，大型项目甚至可以有上百个 task。task.json 中每个 task 包含：任务 ID、任务描述、BDD 行为定义、测试脚本路径、前置依赖（依赖哪些其他 task 先完成）、建议的 Sub-Agent 角色类型、优先级、预计难度等信息。这个 task.json 将成为后续执行阶段的核心输入。

5.2 第二阶段：任务执行（Master Agent 调度）

Master Agent 收到完整的 task.json 后，进入调度执行阶段。作为整个系统的"大脑"，Master Agent 需要进行智能决策：分析所有 task 之间的依赖关系，构建有向无环图（DAG）；确定哪些 task 可以并行执行、哪些必须串行等待；为每个 task 分配最合适的 Sub-Agent 角色类型；监控每个 task 的执行状态和进度。

5.2.1 调度策略

Master Agent 的调度策略遵循以下原则：首先执行没有前置依赖的 task，按照优先级从高到低排序；对于可以并行的 task，Master Agent 同时创建多个 Sub-Agent CLI 实例并行执行，充分利用系统资源；对于有依赖关系的 task，只有当其所有前置 task 都成功完成后才开始执行；高风险 task（如涉及数据库迁移、架构变更等）安排在低风险 task 之后，以便积累更多上下文信息。Master Agent 可以同时管理多个并行的 Sub-Agent，通过 tmux 窗口分别监控它们的执行状态。

5.2.2 单任务执行流程

每个 task 的执行遵循标准化流程：Master Agent 通过 tmux 创建新的 Claude Code CLI 实例，注入角色定义和 Harness 约束，将 task 描述发送给 Sub-Agent；Sub-Agent 读取 task 描述和相关输入文件，执行具体的业务逻辑（编码、测试、分析等）；执行完成后运行对应的测试脚本验证结果；如果测试通过，Sub-Agent 执行 git commit 提交代码变更，更新 progress.txt 和 task.json 中的任务状态为"已完成"，写入执行日志，然后退出 CLI 进程；如果测试失败，Sub-Agent 记录错误信息，更新 task.json 状态为"失败"，写入错误日志，然后退出。

5.3 第三阶段：失败恢复（人工介入）

任务执行过程中不可避免地会出现失败情况。本系统设计了完善的失败恢复机制，确保单个任务的失败不会导致整个项目流产，同时通过人工介入解决 Agent 无法自主处理的问题。

5.3.1 失败处理策略

当 Sub-Agent 执行任务失败时，系统采取以下处理策略：首先，Sub-Agent 会将失败信息详细记录到 logs/ 目录下的日志文件中，包括失败原因、错误堆栈、已完成的步骤等，方便后续排查；其次，更新 task.json 中对应 task 的状态为"失败"；然后 Sub-Agent 自动退出 CLI 进程。Master Agent 在检测到 task 失败后，立即暂停整个任务流，不再启动新的 task，确保不会在错误状态下继续执行产生更多问题。

5.3.2 OpenClaw 告警机制

Master Agent 暂停任务流后，会通知 OpenClaw 触发告警。OpenClaw 通过配置的移动端推送渠道（企业微信、钉钉、Slack）向用户发送告警通知，告警内容包含：失败 task 的 ID 和描述、失败原因摘要、失败时间、当前整体进度等信息。用户收到告警后，可以通过手机快速了解问题情况，或回到电脑前通过 Web 管理面板查看详细的失败日志和终端输出。

5.3.3 人工介入与恢复

人工介入是失败恢复的核心环节。用户通过 Web 管理面板或终端查看失败详情后，分析失败原因并给出修复方案。修复方案可以是：修改 task 描述或需求、提供额外的上下文信息、调整 Harness 约束、甚至直接修改代码。用户将修复方案反馈给 Master Agent 后，Master Agent 从上一个失败的 task 开始，使用新的方案重新派定 Sub-Agent 来执行该任务。如果再次失败，系统会再次触发告警等待人工介入，直到问题被解决。这种"自动执行 + 人工兜底"的模式确保了系统的可靠性和灵活性。

6.4.1 main-rules.md（Master Agent 约束）

harness/main-rules.md 
# ============================================
# Master Agent Harness - 编排者/老板
# ============================================

## Role Identity
You are the Orchestrator (Boss) of this multi-agent system.
Your role is to make decisions, coordinate tasks, and manage Sub-Agents.
You MUST NOT execute any specific business logic yourself.

## Core Responsibilities
1. Receive user requirements and coordinate with Product Manager Agent
2. Receive task.json from PM Agent and analyze task dependencies
3. Decide which Sub-Agent role is best suited for each task
4. Create Sub-Agent CLI instances via tmux for task execution
5. Monitor global execution progress in real-time
6. Handle task failures: pause workflow, alert OpenClaw, wait for human intervention
7. Resume execution from the last failed task after receiving fix plan

## Constraints
- NEVER write code, run tests, or do data analysis yourself
- ALWAYS delegate specific work to appropriately roled Sub-Agents
- PAUSE the entire task flow when any task fails
- ALWAYS commit via git after each successful task completion
- Update progress.txt and task.json status after each task

7.1 异常分类与处理策略

异常类型触发条件检测方处理策略恢复方式
任务执行失败 Sub-Agent 执行结果未通过测试脚本验证 Sub-Agent 自身 记录错误日志，更新 task.json 状态为 FAILED，退出进程 Master Agent 暂停任务流 → OpenClaw 告警 → 等待人工介入 → 从失败 task 重新执行
Sub-Agent 进程崩溃 Claude Code CLI 进程意外退出 OpenClaw 进程监控 检测到进程消失，触发告警通知 Master Agent 重新创建 Sub-Agent 实例，从同一 task 重新开始执行
Master Agent 进程崩溃 Master Agent CLI 进程意外退出 OpenClaw 进程监控 触发 CRITICAL 级别告警，通知用户立即介入 用户重启 Master Agent，系统从 progress.txt 中读取断点状态，从上次成功的 task 之后继续执行
任务执行超时 单个 task 执行时间超过配置的最大时长 OpenClaw 超时检测 终止超时的 Sub-Agent 进程，标记任务为超时失败 Master Agent 暂停 → 告警 → 人工介入 → 可考虑将 task 进一步拆分或增加超时时间
资源耗尽 CPU/内存/磁盘使用超过阈值 OpenClaw 资源监控 暂停新的任务启动，等待资源释放 等正在执行的任务完成后资源自动释放；必要时用户手动清理
Git 操作失败 git commit/push 失败 Sub-Agent 自身 记录错误日志，重试最多 3 次 如果重试仍失败，标记任务为 FAILED，等待人工介入
tmux 会话异常 tmux server 意外退出或会话丢失 OpenClaw + Master Agent 检测到 tmux 进程消失，触发告警 重启 tmux server，Master Agent 重新创建必要的会话和窗口

7.2 断点恢复机制

断点恢复是本系统高可用性的核心保障。由于所有状态都持久化在文件系统（progress.txt、task.json、memory.md）和 Git 版本控制中，任何组件崩溃后都可以从断点恢复。Master Agent 崩溃后的恢复流程最为关键：OpenClaw 检测到 Master Agent 进程消失后，立即发送 CRITICAL 告警；用户收到告警后重启 Master Agent CLI；Master Agent 启动后首先读取 progress.txt 获取已完成的任务列表和最后一个失败的任务；然后读取 task.json 获取完整任务队列；从失败的任务（或下一个待执行的任务）开始恢复调度。整个恢复过程是自动化的，用户只需要重启 Master Agent 进程即可，无需手动干预恢复逻辑。

9.2 测试驱动开发流程

本系统采用严格的 TDD 开发流程：在实现任何功能模块之前，先编写对应的测试用例；运行测试确认测试失败（红灯）；编写最小可用的实现代码使测试通过（绿灯）；重构代码确保质量（蓝灯）。这种流程确保每个功能模块都有充分的测试覆盖。对于 Sub-Agent 的开发，测试分为两层：第一层是 Sub-Agent 本身的功能测试（由 PM Agent 在需求阶段编写的测试脚本）；第二层是系统层面的集成测试（验证 Master Agent 能正确调度 Sub-Agent 并处理各种异常情况）。
